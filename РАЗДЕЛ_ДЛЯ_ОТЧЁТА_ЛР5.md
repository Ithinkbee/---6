## 7. Особенности реализации

Программная реализация диалоговой системы выполнена в виде полностекового веб-приложения, включающего серверную часть (backend), клиентскую часть (frontend) и систему управления базой данных. Ниже приведён перечень используемых инструментов и обоснование их выбора.

### 7.1. Технический стек

| Компонент | Инструмент | Назначение |
|---|---|---|
| Backend-фреймворк | FastAPI (Python 3.12) | Реализация REST API; обработка HTTP-запросов от клиента |
| ORM | SQLModel | Декларативное описание таблиц БД и валидация данных через Pydantic-схемы |
| База данных | PostgreSQL 16 | Хранение музыкальных сущностей (исполнители, альбомы, треки, мероприятия) и истории диалога |
| LLM-оркестрация | LangChain + LangGraph | Управление инструментами ИИ-агента, поддержание памяти диалога, построение ReAct-графа |
| Языковая модель | Groq API (LLaMA 3.3 70B) | Классификация намерений пользователя, извлечение именованных сущностей, генерация ответов |
| Frontend | React 19 + TypeScript | SPA-интерфейс чат-окна, страниц поиска и событий |
| Стилизация | Tailwind CSS + Radix UI | Компонентная система дизайна; адаптивная вёрстка |
| Авторизация | Firebase Authentication | Управление сессиями пользователей (вход через Google) |
| Развёртывание | Docker + Docker Compose | Контейнеризация всех сервисов; воспроизводимый запуск одной командой |

### 7.2. Принцип работы классификации намерений

В реализованной системе классификация намерений осуществляется посредством механизма **tool-calling** большой языковой модели (LLaMA 3.3 70B через Groq API). Система не использует отдельный классификатор; вместо этого модель самостоятельно принимает решение о вызове одного из шести инструментов:

| Инструмент | Соответствующий тип намерения |
|---|---|
| `search_artists` | SEARCH — поиск исполнителей |
| `get_artist_info` | INFO — информация об исполнителе |
| `search_tracks` | SEARCH — поиск треков |
| `recommend_music` | RECOMMEND — рекомендации |
| `get_events` | EVENTS — музыкальные мероприятия |
| `get_genre_info` | INFO — информация о жанре |

Если ни один из инструментов не был вызван (например, запрос «Что ты умеешь?»), система интерпретирует намерение как HELP и отвечает непосредственно. Данный подход реализует неявную классификацию намерений SEARCH / INFO / RECOMMEND / EVENTS / HELP без жёсткого задания шаблонов, что соответствует современным практикам разработки диалоговых систем.

### 7.3. Архитектура ReAct-агента

Обработка каждого запроса выполняется по схеме **ReAct (Reasoning + Acting)** с использованием LangGraph:

1. Пользовательский запрос сохраняется в базе данных и добавляется к истории диалога (до 20 последних сообщений).
2. История передаётся LLM вместе с системным промптом, описывающим роль музыкального ассистента.
3. LLM решает: вызвать инструмент (→ обращение к PostgreSQL) или дать прямой ответ.
4. При вызове инструмента результат возвращается в LLM как контекст для генерации финального ответа.
5. Ответ ассистента сохраняется в базе данных и возвращается клиенту.

Такая архитектура обеспечивает как поддержание контекста многоходового диалога, так и доступ к актуальным структурированным данным из базы.
